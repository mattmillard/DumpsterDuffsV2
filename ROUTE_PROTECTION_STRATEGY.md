# Admin Route Protection Strategy\n\nThis document outlines the authentication and authorization strategy for protecting admin routes in DumpsterDuffs.\n\n## Overview\n\nThe admin section uses a **multi-layer security approach**:\n\n1. **Middleware Layer** (`middleware.ts`) - Session validation on every request\n2. **Layout Layer** (`app/admin/layout.tsx`) - React component-level auth check\n3. **Utility Layer** (`lib/auth/admin.ts`) - Role-based permission checking\n4. **Database Layer** - Row-Level Security (RLS) policies in Supabase\n\n## Layer 1: Middleware (Request-Level)\n\n**File**: `/middleware.ts`\n\n**Responsibility**: Intercept all requests to `/admin` routes and validate session\n\n**Flow**:\n`\nRequest to /admin/dashboard\n    ↓\nIs it /admin/login? → YES → Allow (public route) → Continue\n    ↓ NO\nStart request to /admin route\n    ↓\nCreate Supabase client from cookies\n    ↓\nCheck: Is there a valid session?\n    ↓ YES\nCheck: Does user exist in admin_users table?\n    ↓ YES\nAllow request to proceed\n    ↓ NO (any check fails)\nRedirect to /admin/login with redirect param\n`\n\n**Key Features**:\n- Validates session from Supabase auth cookies\n- Queries `admin_users` table to verify admin status\n- Redirects to login page with redirect URL (user returns to original page after login)\n- Catches all errors and redirects to login for security\n- Runs on every request (fast, cached connections)\n\n**Protected Routes**:\n`\n/admin/*          → Protected (except /admin/login)\n/admin/login      → Public (no auth required)\n/api/*            → Public (manual auth checks in route handlers)\n/booking/*        → Public (customer flow)\n/                 → Public (home page)\n`\n\n## Layer 2: Layout Component (React-Level)\n\n**File**: `/app/admin/layout.tsx`\n\n**Responsibility**: Render component-level auth checks, loading states, and layout structure\n\n**Flow**:\n`\nRender AdminLayout\n    ↓\nOn mount: Call getCurrentAdminUser()\n    ↓\nIs loading? → YES → Show loading spinner\n    ↓ NO\nDid auth check fail? → YES → Show error message\n    ↓ NO\nIs user authenticated? → YES → Render sidebar + header + children\n    ↓ NO\nComponent redirect to /admin/login (backup if middleware fails)\n`\n\n**Key Features**:\n- Redundant check (backup if middleware fails)\n- Display friendly loading/error states\n- Render protected layout (sidebar + header) only for authenticated users\n- Pass admin user context to children\n- Handle Supabase client initialization errors\n\n**States**:\n- **Loading**: Spinner/skeleton while fetching user\n- **Error**: Error message with troubleshooting link\n- **Authenticated**: Full admin interface\n- **Unauthorized**: Redirect to login (shouldn't reach here due to middleware)\n\n## Layer 3: Utility Functions (Helper-Level)\n\n**File**: `/lib/auth/admin.ts`\n\n**Responsibility**: Provide utility functions for auth checks and permissions\n\n**Key Functions**:\n\n### `getCurrentAdminUser()`\n`typescript\nasync function getCurrentAdminUser(): Promise<AdminUser | null>\n`\n- Returns current authenticated admin user with role\n- Returns `null` if not authenticated or not an admin\n- Used by layout.tsx to check auth status\n\n### `adminLogin(email, password)`\n`typescript\nasync function adminLogin(email: string, password: string): Promise<{ user: AuthUser }>\n`\n- Authenticates user with Supabase\n- Returns auth user on success\n- Throws error on failure (display in UI)\n\n### `adminLogout()`\n`typescript\nasync function adminLogout(): Promise<void>\n`\n- Signs out current user\n- Clears session cookies\n- Called from AdminSidebar logout button\n\n### `canUserPerformAction(role, action)`\n`typescript\nfunction canUserPerformAction(role: AdminRole, action: AdminPermission): boolean\n`\n- Checks if user with given role can perform action\n- Uses permission matrix (see below)\n- Example: `canUserPerformAction('dispatcher', 'users:delete')` → false\n\n### `checkPermission(action, throwOnFail = true)`\n`typescript\nasync function checkPermission(action: string, throwOnFail?: boolean): Promise<boolean>\n`\n- Shortcut to get current user role and check permission\n- Auto-fetches current user\n- Throws error if check fails and `throwOnFail = true`\n\n## Layer 4: Database (RLS Policies)\n\n**File**: Supabase project settings\n\n**Responsibility**: Enforce access control at database level\n\n**Policies**:\n\n### Admin Users Table\n`sql\n-- Users can read their own profile\nCREATE POLICY \"admin_users_read_own\" ON admin_users\n  FOR SELECT USING (auth.uid() = id);\n\n-- Users can update their own profile (non-role fields only)\nCREATE POLICY \"admin_users_update_own\" ON admin_users\n  FOR UPDATE USING (auth.uid() = id)\n  WITH CHECK (auth.uid() = id AND role = (SELECT role FROM admin_users WHERE id = auth.uid()));\n`\n\n### Bookings Table\n`sql\n-- Admin can read all bookings\nCREATE POLICY \"bookings_read_admin\" ON bookings\n  FOR SELECT USING (\n    EXISTS (\n      SELECT 1 FROM admin_users\n      WHERE admin_users.id = auth.uid()\n      AND admin_users.is_active = true\n    )\n  );\n\n-- Admin can update bookings\nCREATE POLICY \"bookings_update_admin\" ON bookings\n  FOR UPDATE USING (\n    EXISTS (\n      SELECT 1 FROM admin_users\n      WHERE admin_users.id = auth.uid()\n      AND admin_users.is_active = true\n    )\n  );\n`\n\n**Why RLS at Database Level?**\n- Defense in depth: even if middleware/layout fails, database enforces access\n- API directly queries: any direct DB access (webhooks, cron jobs) respects policies\n- Role consistency: permissions defined once, enforced everywhere\n\n## Permission Matrix\n\nPermissions by role:\n\n| Action | Owner | Admin | Dispatcher |\n|--------|-------|-------|------------|\n| **Bookings** | | | |\n| view | ✅ | ✅ | ✅ |\n| create | ✅ | ✅ | ✅ |\n| update | ✅ | ✅ | ✅ |\n| delete | ✅ | ✅ | ❌ |\n| **Inventory** | | | |\n| view | ✅ | ✅ | ❌ |\n| manage | ✅ | ✅ | ❌ |\n| **Pricing & Zones** | | | |\n| view | ✅ | ✅ | ❌ |\n| edit | ✅ | ✅ | ❌ |\n| **Users** | | | |\n| view | ✅ | ❌ | ❌ |\n| create | ✅ | ❌ | ❌ |\n| delete | ✅ | ❌ | ❌ |\n| **Reports** | | | |\n| view | ✅ | ✅ | ❌ |\n\n**Roles**:\n- **Owner**: Full access, can manage other admins\n- **Admin**: Operational access, can't manage users\n- **Dispatcher**: Limited to booking assignment/tracking\n\n## Implementation Examples\n\n### Example 1: Protect a Route Based on Role\n\n`typescript\n// app/admin/users/page.tsx\nimport { getCurrentAdminUser } from '@/lib/auth/admin';\nimport { redirect } from 'next/navigation';\n\nexport default async function AdminUsersPage() {\n  const user = await getCurrentAdminUser();\n  \n  if (!user || user.role !== 'owner') {\n    redirect('/admin/dashboard');\n  }\n\n  return <div>User Management (Owner Only)</div>;\n}\n`\n\n### Example 2: Check Permission in API Route\n\n`typescript\n// app/api/admin/bookings/route.ts\nimport { getCurrentAdminUser, canUserPerformAction } from '@/lib/auth/admin';\nimport { NextResponse } from 'next/server';\n\nexport async function POST(request: Request) {\n  const user = await getCurrentAdminUser();\n  \n  if (!user || !canUserPerformAction(user.role, 'bookings:create')) {\n    return NextResponse.json({ error: 'Unauthorized' }, { status: 403 });\n  }\n\n  // Process booking creation\n}\n`\n\n### Example 3: Component-Level Permission Check\n\n`typescript\n// components/admin/InventorySection.tsx\n'use client';\n\nimport { useAdmin } from '@/contexts/AdminContext';\nimport { canUserPerformAction } from '@/lib/auth/admin';\n\nexport function InventorySection() {\n  const { user } = useAdmin();\n  \n  const canManage = user && canUserPerformAction(user.role, 'inventory:write');\n  \n  return (\n    <div>\n      <h2>Inventory</h2>\n      {canManage && <button>Add Unit</button>}\n    </div>\n  );\n}\n`\n\n## Flow Diagrams\n\n### Login Flow\n`\nUser visits /admin/login\n    ↓\nMiddleware: route is /admin/login → allow (public)\n    ↓\nDisplay login form\n    ↓\nUser enters email/password\n    ↓\nadminLogin(email, password) called\n    ↓\nSupabase.auth.signInWithPassword()\n    ↓\nValidation success? → YES → \n    Session stored in cookies\n    ↓\n    Redirect to /admin/dashboard (or redirect param)\n    ↓\nValidation fails → NO →\n    Show error message\n    ↓\n    User retries\n`\n\n### Protected Route Access Flow\n`\nUser visits /admin/bookings\n    ↓\nMiddleware checks session:\n    1. Extract session from cookies\n    2. Query admin_users table\n    3. Is user active admin? → YES → Allow\n    4. Is user active admin? → NO → Redirect to /admin/login\n    ↓\nLayout component mounts:\n    1. Call getCurrentAdminUser()\n    2. Render AdminLayout with user data\n    3. User can now access bookings page\n`\n\n### Logout Flow\n`\nUser clicks logout in AdminSidebar\n    ↓\nadminLogout() called\n    ↓\nSupabase.auth.signOut()\n    ↓\nSession cookies cleared\n    ↓\nRedirect to /admin/login\n    ↓\nMiddleware sees no session → redirects to /admin/login\n    ↓\nLogin page displayed\n`\n\n## Session Persistence\n\n**How sessions persist:**\n\n1. User logs in → Supabase creates session tokens\n2. Tokens stored in HTTP-only cookies (secure, not accessible to JS)\n3. Middleware reads cookies on each request\n4. Session persists across page reloads and browser restarts\n5. Session expires after configured TTL (default: 1 hour JWT + 7 days refresh)\n\n**Configuration** (Supabase):\n- Go to **Authentication** → **Policies** → \"JWT expiration limit\"\n- Adjust to desired session length\n- Longer = better UX, shorter = more secure\n\n## Handling Edge Cases\n\n### Case 1: Session Expires While User is Active\n\n`typescript\n// admin_users can refresh auth before expiry\nconst onAuthStateChange = (callback: (user: AdminUser | null) => void) => {\n  const { unsubscribe } = supabase.auth.onAuthStateChange(\n    async (event, session) => {\n      if (event === 'TOKEN_REFRESHED') {\n        // Session auto-refreshed\n        callback(await getCurrentAdminUser());\n      } else if (event === 'SIGNED_OUT') {\n        callback(null);\n      }\n    }\n  );\n  return unsubscribe;\n};\n`\n\n### Case 2: User is Deleted While Still Logged In\n\nMiddleware checks `admin_users` table on every request:\n1. Session valid but user deleted → Middleware detects and redirects to login\n2. Next request to middleware → No session → Login page\n\n### Case 3: User Role Changed\n\n1. Admin's role changed from \"admin\" to \"dispatcher\" in database\n2. User makes request → Middleware queries `admin_users`\n3. User still has valid session, but new role is \"dispatcher\"\n4. Page checks `canUserPerformAction(\"dispatcher\", \"inventory:read\")` → false\n5. Component or layout redirects user appropriately\n\n## Security Best Practices\n\n✅ **DO**:\n- Use HTTPS in production (Supabase enforces this)\n- Keep service role key private (backend only)\n- Validate permissions on both frontend and backend\n- Use RLS policies to enforce access at database level\n- Monitor failed login attempts\n- Expire sessions after reasonable timeout\n- Implement rate limiting on login endpoint\n- Log all admin actions for audit trail\n\n❌ **DON'T**:\n- Expose service role key in frontend code\n- Trust client-side permission checks alone\n- Store passwords in plain text\n- Use weak password requirements\n- Skip middleware checks\n- Disable RLS policies\n- Log sensitive data (passwords, PII)\n\n## Testing the Protection\n\n### Test 1: Unauthorized Access\n`bash\n# Try to access /admin/dashboard without logging in\ncurl http://localhost:3000/admin/dashboard\n# Expected: 307 redirect to /admin/login\n`\n\n### Test 2: Session Validation\n`typescript\n// In browser console:\nawait fetch('/admin/dashboard', {\n  method: 'GET',\n  credentials: 'include' // Include cookies\n});\n// With valid session: 200 OK\n// Without session: 307 redirect to /admin/login\n`\n\n### Test 3: Role-Based Access\n`typescript\n// As dispatcher user:\nawait fetch('/api/admin/users', { credentials: 'include' });\n// Expected: 403 Forbidden (dispatcher can't access user management)\n`\n\n## Monitoring & Debugging\n\n**Enable Supabase Logs:**\n1. Go to Supabase dashboard\n2. **Authentication** → **Auth Monitor**\n3. View all login attempts, failures, session events\n\n**Local Development Debugging:**\n`typescript\n// Add to lib/auth/admin.ts for debugging\nconst DEBUG = process.env.NODE_ENV === 'development';\n\nif (DEBUG) {\n  console.log('Auth check:', { user, role, permissions });\n}\n`\n\n**Common Issues:**\n- \"Unauthorized\" on admin pages → Check middleware.ts matcher config\n- Login form not submitting → Check API keys in .env.local\n- Session expires too quickly → Increase JWT TTL in Supabase settings\n- CORS errors → Check Supabase redirect URLs in project settings\n\n## Future Enhancements\n\n1. **OAuth Integration**: Add Google/GitHub login for admins\n2. **MFA (Multi-Factor Authentication)**: Require authenticator app\n3. **Temporary Links**: Generate temporary login links for new admins\n4. **Session Management**: Admin panel to see/revoke active sessions\n5. **Audit Logging**: Complete action audit trail\n6. **API Keys**: Allow admins to create API keys for integrations\n\n## Summary\n\nThe multi-layer protection strategy ensures:\n\n- ✅ All unauthorized access attempts are blocked at request level\n- ✅ Authenticated access is verified at component and database level\n- ✅ Role-based permissions prevent unauthorized actions\n- ✅ Sessions persist securely across page reloads\n- ✅ Edge cases (expired sessions, deleted users) are handled gracefully\n
