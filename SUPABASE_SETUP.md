# Supabase Authentication Setup Guide

This document provides step-by-step instructions for setting up Supabase authentication for the DumpsterDuffs admin system.

## Prerequisites

- Supabase account (https://supabase.com)
- Node.js 18+ installed locally
- Environment variables configured

## Step 1: Create Supabase Project

1. Go to [https://app.supabase.com](https://app.supabase.com)\n2. Click \"New Project\"\n3. Fill in project details:\n - **Name**: dumpster-duffs-admin (or your choice)\n - **Database Password**: Create a strong password\n - **Region**: Select closest to your users\n4. Wait for project to initialize (5-10 minutes)\n\n## Step 2: Get API Keys\n\n1. In Supabase dashboard, go to **Settings** → **API**\n2. Copy the following keys:\n - **Project URL**: `https://[project-id].supabase.co`\n - **Anon Key (public)**: `eyJhbGc...` (safe to expose in frontend)\n - **Service Role Key (secret)**: `eyJhbGc...` (keep private, backend only)\n\n## Step 3: Configure Environment Variables\n\nCreate `.env.local` in your project root:\n\n`bash\n# Supabase\nNEXT_PUBLIC_SUPABASE_URL=https://[project-id].supabase.co\nNEXT_PUBLIC_SUPABASE_ANON_KEY=eyJhbGc...\nSUPABASE_SERVICE_ROLE_KEY=eyJhbGc...\n`\n\n**Important**: \n- Variables prefixed with `NEXT_PUBLIC_` are exposed to the browser (use anon key)\n- `SUPABASE_SERVICE_ROLE_KEY` is backend only (never expose to frontend)\n- Add `.env.local` to `.gitignore` to prevent leaking credentials\n\n## Step 4: Create Database Tables\n\n### Admin Users Table\n\nIn Supabase SQL Editor, run:\n\n`sql\n-- Create admin_users table\nCREATE TABLE admin_users (\n  id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,\n  email TEXT NOT NULL UNIQUE,\n  full_name TEXT,\n  role TEXT NOT NULL DEFAULT 'admin', -- 'owner', 'admin', 'dispatcher'\n  is_active BOOLEAN DEFAULT true,\n  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),\n  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),\n  last_login TIMESTAMP WITH TIME ZONE\n);\n\n-- Enable RLS\nALTER TABLE admin_users ENABLE ROW LEVEL SECURITY;\n\n-- Create policy: users can read their own profile\nCREATE POLICY \"admin_users_read_own\" ON admin_users\n  FOR SELECT USING (auth.uid() = id);\n\n-- Create policy: admin can update own profile\nCREATE POLICY \"admin_users_update_own\" ON admin_users\n  FOR UPDATE USING (auth.uid() = id)\n  WITH CHECK (auth.uid() = id);\n\n-- Create index for faster queries\nCREATE INDEX idx_admin_users_email ON admin_users(email);\n`\n\n### Bookings Table (for admin to view)\n\n`sql\nCREATE TABLE bookings (\n  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),\n  customer_name TEXT NOT NULL,\n  customer_email TEXT NOT NULL,\n  customer_phone TEXT NOT NULL,\n  size_yards INTEGER NOT NULL,\n  delivery_address TEXT NOT NULL,\n  delivery_date DATE NOT NULL,\n  return_date DATE NOT NULL,\n  total_price DECIMAL(10, 2) NOT NULL,\n  status TEXT DEFAULT 'pending', -- 'pending', 'scheduled', 'in_progress', 'completed', 'cancelled'\n  payment_status TEXT DEFAULT 'unpaid', -- 'unpaid', 'paid', 'refunded'\n  notes TEXT,\n  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),\n  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()\n);\n\nALTER TABLE bookings ENABLE ROW LEVEL SECURITY;\n\n-- Admin can read all bookings\nCREATE POLICY \"bookings_read_admin\" ON bookings\n  FOR SELECT USING (\n    EXISTS (\n      SELECT 1 FROM admin_users\n      WHERE admin_users.id = auth.uid()\n      AND admin_users.is_active = true\n    )\n  );\n`\n\n## Step 5: Set Up Authentication\n\n### Email/Password Authentication\n\n1. In Supabase dashboard, go to **Authentication** → **Providers**\n2. Make sure **Email** provider is enabled\n3. Go to **Email Templates** → **Confirm signup**\n4. Configure email settings for production\n\n### Create Initial Admin User\n\nIn Supabase SQL Editor:\n\n`sql\n-- Use Supabase auth API to create user (or create manually in Auth panel)\n-- Then insert into admin_users table\n\nINSERT INTO admin_users (id, email, full_name, role, is_active)\nVALUES (\n  'paste-uuid-from-auth-user-here',\n  'admin@example.com',\n  'Admin User',\n  'owner',\n  true\n);\n`\n\n**OR** use the Supabase Auth panel:\n\n1. Go to **Authentication** → **Users**\n2. Click **Add User**\n3. Enter email and password\n4. Create user\n5. Copy the user ID and insert into `admin_users` table (see SQL above)\n\n## Step 6: Configure Next.js for SSR\n\nThe app uses Supabase SSR utilities for secure server-side authentication.\n\n**Key files already configured:**\n- `/lib/supabase/client.ts` - Browser client\n- `/middleware.ts` - Route protection\n- `/lib/auth/admin.ts` - Auth utilities\n- `/app/admin/layout.tsx` - Auth check on render\n\n## Step 7: Test the Authentication Flow\n\n1. Start the dev server:\n `bash\n   npm run dev\n   `\n\n2. Navigate to `http://localhost:3000/admin/login`\n\n3. Test login:\n - Email: `admin@example.com` (or your created user)\n - Password: Your password\n - Click \"Sign In\"\n\n4. On successful login:\n - Redirected to `/admin/dashboard`\n - Session persisted in cookies\n - User can navigate admin pages\n\n5. Test logout:\n - Click user profile → Logout in AdminSidebar\n - Redirected to `/admin/login`\n\n## Step 8: Implement Additional Features\n\n### User Management\n\nCreate admin controls to add/remove users:\n\n`typescript\n// Example: Create new admin user\nimport { supabase } from '@/lib/supabase/client';\n\nasync function createAdminUser(email: string, password: string, role: 'admin' | 'dispatcher') {\n  // Create auth user\n  const { data: authUser, error: authError } = await supabase.auth.admin.createUser({\n    email,\n    password,\n    email_confirm: true\n  });\n\n  if (authError) throw authError;\n\n  // Create admin_users record\n  const { error: dbError } = await supabase\n    .from('admin_users')\n    .insert({\n      id: authUser.user.id,\n      email,\n      role,\n      is_active: true\n    });\n\n  if (dbError) throw dbError;\n}\n`\n\n### Session Persistence\n\nSupabase automatically persists sessions in cookies via the middleware. No additional configuration needed.\n\n### Role-Based Access Control (RBAC)\n\nThe `/lib/auth/admin.ts` file includes a permissions matrix:\n\n`typescript\nconst PERMISSIONS = {\n  owner: ['all'],\n  admin: ['bookings:read', 'bookings:write', 'inventory:read', 'inventory:write', 'zones:read', 'zones:write'],\n  dispatcher: ['bookings:read', 'bookings:write']\n};\n`\n\nTo implement role-based routes, update middleware matcher:\n\n`typescript\nif (pathname === '/admin/users' && userData.role !== 'owner') {\n  return NextResponse.redirect(new URL('/admin/dashboard', request.url));\n}\n`\n\n## Step 9: Production Deployment\n\n### Environment Variables for Production\n\nSet these in your hosting provider (Vercel, Netlify, etc.):\n\n`\nNEXT_PUBLIC_SUPABASE_URL=your-project-url\nNEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key\nSUPABASE_SERVICE_ROLE_KEY=your-service-role-key\n`\n\n### Supabase Project Settings\n\n1. Go to **Authentication** → **Policies** → Add authorized redirect URLs:\n - Development: `http://localhost:3000`\n - Production: `https://yourdomain.com`\n\n2. Enable email verification in **Authentication** → **Policies**\n\n3. Configure SMTP for production emails in **Authentication** → **Email Templates**\n\n## Troubleshooting\n\n### \"Invalid API key\" Error\n\n- Verify `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY` in `.env.local`\n- Check keys haven't been rotated in Supabase dashboard\n- Restart dev server after changing `.env.local`\n\n### \"User not found\" Error on Login\n\n- Ensure user exists in Supabase **Authentication** → **Users**\n- Check `admin_users` table has corresponding record\n- Verify email is correct (case-sensitive in some cases)\n\n### Middleware Not Redirecting to Login\n\n- Ensure `middleware.ts` is in project root (not in `/app`)\n- Verify matcher config includes `/admin` routes\n- Check browser cookies are enabled\n- Clear browser cookies and try again\n\n### Session Expires Too Quickly\n\n- Go to Supabase **Authentication** → **Policies**\n- Adjust \"JWT expiration limit\" (default 3600 seconds)\n- Increase \"Refresh token rotation\" timeout for longer persistence\n\n## Testing Credentials (Development Only)\n\nFor quick testing without manual signup:\n\n**Email:** `admin@example.com`\n**Password:** `TestPassword123!`\n\nThese are referenced in `/app/admin/login/page.tsx` as a development note.\n\n## Security Checklist\n\n- [ ] `.env.local` added to `.gitignore`\n- [ ] Service Role Key never exposed to frontend\n- [ ] Row-Level Security (RLS) enabled on all tables\n- [ ] Auth policies restrict user access appropriately\n- [ ] Email verification enabled for new users (production)\n- [ ] Strong password requirements enforced\n- [ ] Session timeout configured appropriately\n- [ ] CORS and redirect URLs whitelisted\n- [ ] Database backups enabled in Supabase\n- [ ] Audit logs monitored for suspicious activity\n\n## Next Steps\n\n1. **Implement Stripe Payments** -
